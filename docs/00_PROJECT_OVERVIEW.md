# Sunken Fortune - Project Overview

## Project Identity

| Field | Direction |
| --- | --- |
| Game name | Sunken Fortune |
| Genre | PvE-first pirate naval action RPG / MMORPG-lite |
| Primary experience | Active naval combat |
| Platform target | Browser-first, expandable later |
| Commercial intent | Portfolio-quality foundation suitable for future commercial expansion |
| Multiplayer posture | Future-facing, not MVP-critical |

Sunken Fortune is a legally original pirate naval action RPG inspired by the broad appeal of classic browser pirate MMOs, without copying their protected creative expression. The game centers on active ship combat, long-term captain progression, meaningful ship milestones, and a carefully tuned economy.

## Product Summary

Players captain a pirate vessel through dangerous waters, fighting enemy ships, earning gold and materials, upgrading their captain and crew, crafting better gear, and eventually unlocking a small set of emotionally meaningful ships. The core fantasy is not owning dozens of disposable vessels. It is earning a better ship, equipping it well, mastering its combat profile, and using it to push into harder zones and bosses.

## Core Experience

Active naval combat should be the strongest and most satisfying way to progress. The player should make decisions around:

- Movement and spacing.
- Broadside angles.
- Reload timing.
- Ammunition choice.
- Cannon and module builds.
- Hull condition and repair timing.
- Enemy behavior and boss mechanics.

## Progression Sources

Player progression may come from:

- Ships.
- Cannons.
- Ammunition.
- Modules.
- Captain upgrades.
- Crew upgrades.
- Talents.
- Crafting.
- Rare drops.
- Bosses.
- Zones.
- Future social and cooperative systems.

## MVP Positioning

The MVP should prove the core loop before expanding into full MMO features. PvP, guilds, global chat, co-op, live events, trading, and large-scale social systems are future expansions, not first-build requirements.

## Documentation Purpose

This documentation foundation exists to:

- Define the game identity before implementation begins.
- Prevent accidental copying from existing games.
- Protect the economy and progression design from early shortcuts.
- Establish MVP boundaries.
- Clarify offline progression through Autonomous Voyage.
- Define save-system expectations before economy code exists.
